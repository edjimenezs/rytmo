# Domain Pitfalls: Sports Nutrition / Fueling Recommendation Apps

**Domain:** Endurance athlete daily fueling engine
**Project:** RytMo MVP
**Researched:** 2026-03-22
**Confidence:** HIGH (based on direct codebase analysis + domain knowledge)

---

## Critical Pitfalls

Mistakes that cause rewrites, abandoned MVPs, or products athletes stop using after one week.

---

### Pitfall 1: Two Parallel Nutrition Stacks Producing Different Outputs

**What goes wrong:**
The codebase currently has TWO separate nutrition recommendation paths that produce different types of output:

- `app/api/agents/nutrition-plan` — returns **macros + generic text suggestions** (legacy, LLM-first)
- `app/api/daily-plan` + `lib/nutrition/engine.ts` — returns **concrete foods by moment** (newer, rule-first)

Both routes exist. Both are connected to UI components. When the plan page loads, it may silently serve either path depending on which component renders first or which route the frontend calls. The athlete sees recommendations but has no way to know which engine generated them, or whether they are consistent.

**Why it happens:**
Scope creep during development. Someone built the agent-based approach first (legacy `/api/agents/nutrition-plan`), then a better rule-based engine was built in parallel without retiring the first. The old one still gets called from `components/dashboard/NutritionAgentPanel.tsx`.

**Consequences:**
- Inconsistent UX: one panel shows macros, another shows foods — the same athlete, the same day
- Feedback loop breaks: athlete rates a plan generated by the agent, not the engine. You have no idea which system's output they evaluated
- Impossible to improve: you don't know which recommendation drove which outcome

**Warning signs:**
- Multiple API routes that return "nutrition plan" data with different shapes
- Frontend components importing from different `/api/agents/` vs `/api/daily-plan/` endpoints
- `NutritionPanel.tsx` and `NutritionAgentPanel.tsx` both existing with similar purposes

**Prevention:**
Delete `/api/agents/nutrition-plan` and `NutritionAgentPanel`. Route ALL nutrition plan requests through `daily-plan` + `engine.ts`. Single source of truth. The agent-based endpoint outputs macros as primary data — that's the anti-pattern you're already trying to avoid.

**Phase:** Phase 1 (refactor foundation) — do this before any other nutrition work.

### Pitfall 2: Showing Macros as Primary Output

**What goes wrong:**
The legacy nutrition agent (`/api/agents/nutrition-plan`) returns `{ calories, protein, carbs, fat }` as the primary response. If any UI component renders this, the athlete sees numbers. Numbers require interpretation. Interpretation requires nutrition literacy. Most endurance athletes don't have it, and even those who do don't want to do math before every meal.

This is the **central anti-feature** of apps like MyFitnessPal applied to sport: the output requires the user to do the last mile of work (translate "210g carbs" into "what should I eat right now").

**Why it happens:**
Macros are easy to compute (simple heuristics) and look "scientific." They satisfy the builder's desire to show precision. But they transfer cognitive load to the user at exactly the wrong moment — pre-workout, when they're trying to get out the door.

**Consequences:**
- Athletes disengage after 3-5 days because every recommendation requires interpretation
- The "actionable" core value proposition is lost
- Feedback ratings become meaningless: athlete rates "good" but can't remember what they actually ate

**Warning signs (already present in codebase):**
```
// lib/agents/nutrition-plan/route.ts line 43
const protein = Math.round(1.8 * 70); // 70kg estimado
```
Hardcoded weight, macro output as primary return value. This is the pitfall already manifested.

**Prevention:**
Macros live in `catalog.ts` on each food item but are hidden from UI. The `engine.ts` + `buildNutritionPlan` approach correctly surfaces food names and portions as primary content. Keep it this way. If macros appear in the UI, they must be secondary (collapsed, optional detail) — never the headline.

**Phase:** Phase 1 (kill macro-first output). Phase 2 (validate food-first UX with personal use).

### Pitfall 3: Check-in Friction Above the 60-Second Threshold

**What goes wrong:**
The current `CheckinForm.tsx` has 10 input fields:
1. Fecha (date picker)
2. Sueño horas (number)
3. Duración entrenamiento (number)
4. Sueño calidad (select 1-5)
5. Fatiga (select 1-5)
6. Hambre (select 1-5)
7. Estrés (select 1-5)
8. Tipo de entrenamiento (free text)
9. Intensidad (select: Low/Moderate/High)
10. Momento del día (select: Morning/Afternoon/Evening)
11. Notas (textarea)

That is 11 fields. The PROJECT.md target is sub-60s. On mobile, 11 fields is 3-4 minutes. Friction compounds: if the athlete skips one day because it took too long, they skip again the next day. After a week the habit is broken.

**Why it happens:**
Engineers optimize for data completeness. Every field captures "useful signal." The temptation is to ask for everything upfront, deferring UX work.

**Consequences:**
- Sub-60s target is impossible as currently designed on mobile
- Free-text `trainingType` field produces inconsistent data that can't be reliably used in recommendations
- `notes` field is never read by the engine — it's captured but doesn't influence output

**Warning signs:**
- The `notes` field payload goes to DB but no recommendation logic reads it
- `trainingType` is a free text string, not a controlled enum, making downstream matching unreliable
- The intensity field is in English ("Low", "Moderate", "High") but the app is Spanish-first

**Prevention:**
Strip to 5 fields maximum. The minimum viable check-in for a fueling recommendation is:
- Training type (tap, not type — 5 presets: Rodada Z2, Intervalos, Natación, Carrera, Descanso)
- Duration (one number)
- Intensity (already there, but 3 taps not a select)
- Sleep quality (1 fast tap: malo/regular/bueno)
- [Optional] Fatigue (same)

Everything else can be derived from Strava if connected. The notes field should only exist in the feedback form, not the check-in.

**Phase:** Phase 1 (strip check-in to 5 fields). Phase 2 (validate timing with real use).

### Pitfall 4: AI for Logic, Not for Phrasing

**What goes wrong:**
The existing `/api/agents/nutrition-plan` sends this prompt to GPT-4o-mini:
```
Eres un nutriólogo deportivo. Con estos datos [...] sugiere macros ajustados y 3 recomendaciones prácticas.
```
This asks the AI to do TWO things: (1) determine what foods/macros are appropriate and (2) phrase the recommendations. When AI does the logic, recommendations become inconsistent, unpredictable, and untestable. You can't debug "why did GPT recommend X today but Y yesterday with the same inputs."

**Why it happens:**
LLM APIs make it tempting to outsource all reasoning. "Just ask the AI" is faster than building rule logic, so the MVP starts with an all-AI approach. But LLMs are non-deterministic and have no domain memory — they don't know this athlete trained 4 days in a row.

**Consequences:**
- Recommendations feel arbitrary because they ARE arbitrary — same input, different output
- Cannot improve the engine by adding rules; must prompt-engineer instead
- When AI call fails, entire recommendation fails (no graceful fallback)
- `gpt-4o-mini` with 800 token limit produces truncated or invalid JSON silently

**Warning signs (already in codebase):**
```typescript
// nutrition-plan/route.ts - AI generates macros directly
const aiJson = JSON.parse(aiText); // no schema validation
if (aiJson?.summary) { summary = aiJson.summary ... } // trusts AI output blindly
```
The try/catch around JSON.parse silently falls back to defaults — athlete gets stale data with no indication.

**Prevention:**
The `engine.ts` architecture is correct: deterministic rule logic decides which foods appear. AI is only used to rephrase the output into natural language. This is the "AI for phrasing, not logic" pattern. Keep it. Remove the agent endpoint that inverts this.

The AI call should look like:
```
Given: training=ciclismo_z2, duration=90min, dayType=moderate, foods=["Marraqueta + palta + huevo", "Cazuela de vacuno"]
Task: Write 2 sentences in informal Spanish explaining why these foods today. Be concrete. No macros.
```
Not: "Decide what to eat."

**Phase:** Phase 1 (kill AI-logic endpoint). Phase 2 (add AI phrasing layer to engine output).

### Pitfall 5: Strava Sync Silently Failing Without User Notification

**What goes wrong:**
From `CONCERNS.md`: token refresh returns `null` on failure with only a `console.error`. The daily plan endpoint calls `getDailyLoads()` which queries `TrainingActivity`. If Strava sync failed 3 days ago and the athlete doesn't know, today's recommendation is based on stale/incomplete training history. The engine calculates ATL/CTL/ACWR from whatever is in the DB — which may be missing the last 3 days of training.

The athlete sees a "rest day" recommendation when they actually did a high-intensity interval session yesterday that never synced.

**Why it happens:**
Integration failures are treated as infrastructure problems, not UX problems. Error goes to logs. Athlete gets wrong recommendation.

**Consequences:**
- Wrong fueling for actual training load
- Athlete loses trust in the app within first week ("it recommended I eat light but I had just done a hard ride")
- Impossible to diagnose: athlete can't tell if recommendation is wrong because of bad logic or bad data

**Warning signs (already in codebase):**
```typescript
// lib/strava/utils.ts
.catch((err) => {
  console.error('Token refresh failed', err);
  return null; // silent failure
});
```
And in `load.ts`: no validation that the loads array covers recent days — gaps in data are invisible.

**Prevention:**
- Store `lastSuccessfulSync` timestamp on `StravaIntegration`
- In `daily-plan` endpoint: if `lastSuccessfulSync` is > 2 days ago, add a banner in response: `staleSync: true`
- UI shows: "Tus datos de Strava pueden estar desactualizados. Sincronizar ahora."
- Never serve a recommendation without knowing the freshness of its input data

**Phase:** Phase 1 (add staleSync flag to daily-plan response + UI warning). Critical for trust.

### Pitfall 6: Ignoring Check-in Data in the Recommendation Engine

**What goes wrong:**
The athlete fills out the check-in (fatigue: 5/5, sleep: 4h, stress: 5/5). Then calls `/api/daily-plan`. The `buildNutritionPlan` function receives `planEntry` + `loads` but NEVER reads the check-in data. The plan is identical whether the athlete slept 4h with high stress or slept 9h fresh.

This is the most fundamental failure for a check-in based product: the check-in is disconnected from the output.

**Why it happens:**
Two systems built independently. `engine.ts` was built against `TrainingPlanEntry`. `checkin/route.ts` saves data to `DailyCheckin`. Nobody wired them together yet.

**Consequences:**
- The check-in feels like useless busywork to the athlete
- The product's core loop is broken: check-in → recommendation → feedback is the full cycle, but the middle step ignores the first
- Athlete asks "why did I fill this out?" and stops doing check-ins

**Warning signs (already in codebase):**
```typescript
// daily-plan/route.ts
const planEntry = await findTrainingPlanEntryForDate(userId, normalizedDate);
const planResponse = buildNutritionPlan({ planEntry, loads });
// DailyCheckin is never queried in this flow
```

**Prevention:**
`getDailyPlan` must fetch `DailyCheckin` for today and pass it into `buildNutritionPlan`. Add fatigue modifier: if `fatigue >= 4`, downgrade `requiresIntraFuel`, boost recovery foods. If `sleepHours < 6`, prioritize sleep-supporting dinner foods. These rules are simple, testable, and make the check-in feel consequential.

**Phase:** Phase 1 (wire checkin into engine). The single highest-value connection to make.

## Moderate Pitfalls

### Pitfall 7: Food Catalog Too Small to Cover All Moments + Focus Combinations

**What goes wrong:**
The current `catalog.ts` has 12 items. The `pickFoods()` function filters by `moment` AND `focus`. Some combinations return 0 results and fall back to `.slice(0, 2)` of all foods for that moment — ignoring focus entirely. A "performance + recovery" post-workout recommendation might fall back to the maintenance yogurt parfait.

**Warning signs:**
- `pickFoods` has an explicit fallback: `return foodCatalog.filter(o => o.moment === moment).slice(0, 2)`
- Catalog has no snack items with `performance + recovery` focus
- Catalog has no intraWorkout items with `recovery` or `maintenance` focus

**Prevention:**
Before expanding features, expand catalog to 40 items minimum with intentional coverage of all `moment × focus` combinations. Map coverage as a matrix: `moment (5) × focus (4)` = 20 cells. Aim for 2+ items per cell. 12 items cannot cover this without constant fallback.

**Phase:** Phase 1 (catalog expansion to ~40 items). Required before the engine produces reliable output.

### Pitfall 8: Hardcoded Athlete Profile Values

**What goes wrong:**
```typescript
// load.ts
const DEFAULT_FC_REPOSO = 50;
const DEFAULT_FC_MAX = 185;

// nutrition-plan/route.ts
const protein = Math.round(1.8 * 70); // 70kg estimado
```

TSS estimation uses the same FC values for every athlete. For a lightweight cyclist with FC_reposo=42 and FC_max=178, the ACWR calculation is off by ~15-20%. The nutrition macro calculation assumes 70kg regardless of actual weight.

**Warning signs:**
From `CONCERNS.md`: "Training profile returns hardcoded defaults instead of reading from database Profile table."

**Prevention:**
Add `fcMax`, `fcReposo`, `weight` to the `Profile` schema (already planned in CONCERNS.md). Pass profile into `getDailyLoads()` and nutrition calculations. For founder-only MVP, hardcode the founder's actual values in the Profile seed rather than keeping generic defaults.

**Phase:** Phase 1 (seed real profile values). Phase 2 (add profile edit UI if validating with others).

### Pitfall 9: Feedback Loop Has No Closing Mechanism

**What goes wrong:**
`FeedbackForm.tsx` captures energy, hunger, performance, digestion (1-5 each). This data goes to `DailyFeedback` in the DB. But nothing reads this feedback to adjust future recommendations. The loop is: collect → store → never use.

**Warning signs:**
No code in `engine.ts` or `daily-plan` route queries `DailyFeedback`. The feedback is inert data.

**Prevention:**
For MVP, at minimum: if energy feedback was 2 or below for 2 consecutive days, flag this in the next day's plan context as "persistent low energy." Even a simple rule closes the loop. The longer the feedback stays unread, the less incentive the athlete has to fill it out.

**Phase:** Phase 2 (basic trend reading). Not required for first 3-day personal validation, but needed before any feedback about the app to others.

### Pitfall 10: Timezone Ambiguity Breaking "Today's Plan"

**What goes wrong:**
The `normalizeDate()` function uses `setUTCHours(0, 0, 0, 0)` — this normalizes to midnight UTC. Lalo is in Santiago (UTC-3 or UTC-4 depending on DST). A check-in submitted at 11pm Santiago time is `02:00 UTC next day`, so `normalizeDate()` stores it as tomorrow. Next morning when fetching today's plan, the check-in from last night is "tomorrow's check-in" and today's slot is empty.

**Warning signs:**
```typescript
candidate.setUTCHours(0, 0, 0, 0); // UTC midnight, not local midnight
```
This pattern appears in both `checkin/route.ts` and `daily-plan/route.ts`.

**Prevention:**
Store dates as UTC midnight consistently, but derive "today" from the client's local timezone (send timezone offset in the request), not the server's clock. Alternative: always send `date: YYYY-MM-DD` from the client (which `CheckinForm` already does with `new Date().toISOString().slice(0,10)`) and trust that string rather than re-normalizing. The current form already does this correctly — the bug is in the server re-normalizing to UTC.

**Phase:** Phase 1 (verify timezone handling before extended personal use).

## Minor Pitfalls

---

### Pitfall 11: Intensity Field in English in Spanish-First App

**What goes wrong:**
`intensity` options in `CheckinForm`: `['Low', 'Moderate', 'High']`. The app is Spanish-first. These get stored in DB as English strings and compared downstream. If anyone ever adds Spanish labels as display aliases, the DB values won't match.

**Prevention:** Store as controlled enum values in Spanish from day one: `['baja', 'moderada', 'alta']`. Simple change, expensive to migrate later.

**Phase:** Phase 1 (pre-migration, while DB is fresh).

---

### Pitfall 12: Plan Generation on Every GET Request

**What goes wrong:**
`/api/daily-plan GET` generates a full plan on every page load AND upserts it to the DB. If the athlete opens the app 5 times in a day, they generate 5 plans. The last one overwrites all previous. If they open it at 6am (pre-workout) and again at 9pm (dinner), the dinner-time load might generate a different plan if ACWR shifted due to activity sync.

**Prevention:**
Generate plan once per day (on first load or after checkin save). Cache the result. Allow explicit "regenerate" action. Don't overwrite a plan that was already shown to the user without their intent.

**Phase:** Phase 2 (after confirming single-athlete use doesn't hit this edge case in practice).

---

### Pitfall 13: Scope Creep Vectors Already Present in Codebase

**What goes wrong:**
The codebase contains live code for coaches, nutritionists, medical document analysis, and TrainingPeaks/Garmin integrations. All of this is OUT OF SCOPE for the MVP. But it will be tempting to "just leave it running" because it already exists. Each of these dormant features creates maintenance surface area, potential bugs, and cognitive overhead when navigating the codebase.

**Prevention:**
Explicitly stub or remove non-MVP routes in the refactor phase. At minimum: disable role-based routing to CoachDashboard and NutritionistDashboard. Redirect all non-athlete roles to a "coming soon" page. Don't remove the DB schema (preserves future path) but gate the UI.

**Phase:** Phase 1 (scope gate — before building new features, remove paths to dead features).
